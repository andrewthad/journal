# Better Patching in Packlab

The packlab project, where I have tried out several compression schemes,
has an missed opportunity for saving space. Currently, patching is handled
by providing a patch mask and a list of patch elements. The patch mask is
expected to have a population count equal to the number of patch elements.

This patch mask wastes spaces, typically an entire 64-bit word. It is
possible to use one of the keys instead, probably the key represented
by a sequence of ones.

* For bitpacked data, this means that we would trade away the ability
  to represent the highest number
* For dictionaries, it's the same. A 4-bit key could only encode
  15 elements instead of 16.

It's not clear that this is always going to be a big win. Most of the
64-elements in my test netflow data require between 80 and 120 bytes,
so an 8-byte improvement will be nice. There are some smaller
dictionaries that I'm a little worried about, but it might work
out fine. For bitpacked data, I expect that this will always
improve things.

The problem is, how do we reconstruct the patch mask? We can right
shift-then-and in a loop by increasing powers of two (1, 2, 4) and
then perform bit-extract at the end to grab the low bit.
