# Claude for Bit Projection Algorithm

I have been struggling to solve a problem, and I was able to get some help
from Claude (Opus 4.6). Here's the prompt I gave it:

> I would like to perform multiple bit projections using a 64-bit
> word. Each index into the target bitvector is 2 bits, so the target
> bitvector itself is only 4 bits, and the vector of indices has
> 32 2-bit indices. I do not want to use any instructions that require
> XMM, YMM, or ZMM registers. The result should be a 32-bit word.
> Assume that BMI2 is available.

Here's the answer that it gave:

> This is a parallel 4-to-1 bit mux problem. The key insight: decompose
> each 2-bit index into its high and low bits using PEXT, then evaluate
> a two-level mux tree using standard bitwise ops (with ANDN from BMI1,
> which comes with BMI2).
>
> For each position i, you want result[i] = target[2·hi[i] + lo[i]].
> Separate the 32 low-bits and 32 high-bits of the indices via two
> PEXT operations, broadcast each target bit to a 32-bit mask, then
> compute a two-level multiplexer:

    #include <stdint.h>
    #include <immintrin.h>
    
    // indices: 64-bit word containing 32 × 2-bit indices (index i at bits [2i+1 : 2i])
    // target:  4-bit bitvector (only low 4 bits used)
    // returns: 32-bit word where bit i = target[index_i]
    uint32_t bit_project(uint64_t indices, uint32_t target) {
        // Separate low bit (even positions) and high bit (odd positions) of each 2-bit index
        uint32_t lo = (uint32_t)_pext_u64(indices, 0x5555555555555555ULL);
        uint32_t hi = (uint32_t)_pext_u64(indices, 0xAAAAAAAAAAAAAAAAULL);
    
        // Broadcast each target bit to a full 32-bit mask (0x00000000 or 0xFFFFFFFF)
        uint32_t t0 = -(target & 1u);
        uint32_t t1 = -((target >> 1) & 1u);
        uint32_t t2 = -((target >> 2) & 1u);
        uint32_t t3 = -((target >> 3) & 1u);
    
        // First mux level: select by lo bit
        //   mux0 = lo ? target[1] : target[0]   (for when hi = 0)
        //   mux1 = lo ? target[3] : target[2]   (for when hi = 1)
        uint32_t mux0 = (~lo & t0) | (lo & t1);
        uint32_t mux1 = (~lo & t2) | (lo & t3);
    
        // Second mux level: select by hi bit
        return (~hi & mux0) | (hi & mux1);
    }

And then it gives an analysis of the instruction count. What's interesting
is that during the reasoning phase, it described a different approach (similar
to the one that I had originally considered) but then abandoned it for this
approach instead. I did not think about solving this with a mux.

I have given Claude another question to try to guide me to a specific
solution instead:

> Please write a C function. The argument is a 32-bit word. The bits
> in the word should be interpreted as 16 2-bit unsigned integers.
> The output should be a 64-bit word. The output word is treated
> as 16 4-bit nybbles. Each nybble in the output should have one
> bit set to 1 and the other 3 bits are set to 0. The bit set to 1
> is dictated by the corresponding 2-bit integer from the argument.
> It is fine to use BMI2 instructions. Do not use any SIMD extensions.

Here is the response that it gives:

    #include <stdint.h>
    #include <immintrin.h>
    
    uint64_t bits_to_onehot(uint32_t input)
    {
        /* Spread 16 packed 2-bit values into the low 2 bits of 16 4-bit nibbles */
        uint64_t spread = _pdep_u64(input, 0x3333333333333333ULL);
    
        /* Isolate bit 0 and bit 1 of each nibble */
        uint64_t b0  = spread & 0x1111111111111111ULL;
        uint64_t b1  = (spread >> 1) & 0x1111111111111111ULL;
    
        /* Per-nibble inversions */
        uint64_t nb0 = b0 ^ 0x1111111111111111ULL;
        uint64_t nb1 = b1 ^ 0x1111111111111111ULL;
    
        /* One-hot encode: set bit (b1*2 + b0) within each nibble */
        return (nb1 & nb0)          /* bit 0: value == 0 */
             | ((nb1 & b0)  << 1)  /* bit 1: value == 1 */
             | ((b1  & nb0) << 2)  /* bit 2: value == 2 */
             | ((b1  & b0)  << 3); /* bit 3: value == 3 */
    }

With this explanation:

> * `_pdep_u64` (BMI2) deposits the 32 input bits into the
>   lower 2 bits of each 4-bit nibble in a 64-bit word,
>   spacing them out from ..ba packing to ..ba.... nibbles.
> * `b0 / b1` isolate the low and high bit of each 2-bit value
>   (one bit per nibble).
> * The four AND/OR/shift lines compute the standard 2-to-4 binary decoder
>   logic in parallel across all 16 nibbles — each term is nonzero only
>   when the 2-bit value matches that term's index, producing exactly
>   one set bit per nibble at the correct position.

# Things I Learned

I've not confirmed whether either of these code samples do what they
claim to do. But this process introduced me to two important concepts
that I was not able to find by querying this type of problem with a
search engine:

1. The term "one-hot encoding". This is a term with a commonly understand
   meaning, but I could not plug the meaning into a search engine and get
   the term.
2. Multiplexing. I already knew what muxtiplexers were.
