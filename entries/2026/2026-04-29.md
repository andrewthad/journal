# Different Compression for Different Types of Integers

I have been thinking about integer encoding as a "one size fits all" thing,
and I've realized that it probably does not make sense to do this. Byte/packet
counts and IP addresses are just very different kinds of data. Additionally,
the type of data should always be known in advance.

# Compressing Byte Counts

What's a way to do this that doesn't involve patching? We could just directly
encode integers with something like LEB128. The one thing about this data type's
distribution that we really want to take advantage of is that integers closer
to zero occur more often. LEB128 would actually work. What are its drawbacks?

* Vectorized element access is difficult or impossible
* It cannot compress elements down to anything smaller than one byte
* Random access requires scanning
* It's not possible to treat raw 64-bit elements as encoded data

And what's good about it?

* The block size does not impact the compression ratio
* Everything is byte aligned (no padding)
* Upcasting 32-bit data to 64-bit data is a no-op.

There are variants of LEB128 that move the control bits all to the front or
that move the control bits out of the data entirely. Moving the control bits
out of the data is the strategy most likely the result in something that
supports vectorized access. StreamVByte is the most straightforward approach
to do this.
