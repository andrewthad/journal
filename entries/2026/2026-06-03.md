# Parallelized Equality Test for All Bitwidths

I've been thinking about how to have a single code path that accelerates
the extremely common case of testing for equality against a single value.
There are two steps here:

1. Compare everything byte by byte, returning a bitmask
2. Reduce this bitmask, grouping bit blocks with conjunction

The first step is easy. It's just XOR. AVX2 has a XOR that works on
256-bit registers: `_mm256_xor_si256` (`vpxor`). You need `_mm256_cmpeq_epi8` (`vpcmpeqb`)
to check all these against 0x00 (build all zeros register outside of the loop).
And then `_mm256_movemask_epi8` (`vpmovmskb`) takes the high bit from each of
these and gives you a 32-bit word in a GPR. There's probably some kind of
trick here that involves using `_mm256_testz_si256` (`vptest`) to try to
short circuit when there are no matches. Perfect, that's step one.

Step 2 can be broken down into: (A) Reduce groups and (B) extract bits. Step B
is just an application of the BMI2 bit-extract instruction. Step A is more
difficult. Claude Opus 4.8 offers this strategy:

    for (int s = 1; s < group_size; s <<= 1) {
        x &= x >> s;
    }

This works when `group_size` is a power of two. It processes `x` in a logarithmic
number of steps. When the `group_size` is 16, it takes 4 steps to get the 32 bits
collapsed down to 2 bits.

I find this strategy unsatisfying. Interestingly, I've noticed that Claude's
emergent behavior is to fixate on this. In the same way, it fixates on
reminding the user that BMI2 instructions are microcoded in AMD Zen 1 and
Zen 2. And on multiple occasions, it has recommended a defective bit twiddling
hack where you add 0x01010101 to a number, but it often discovers as it
recommends it that there's an overflow problem between the bytes. (Although
sometimes it doesn't notice that at all.) Of course, the good thing is
that this particular bit twiddling hack can be made to work. It just needs
the high to be zero.

The good thing is that there is a better way to accomplish this. First build
two group vectors, high hot and low hot, based on the group size. For example,
at size 4:

* High hot: 1000 1000 1000 1000 1000 1000 1000 1000 1000 1000 1000 1000 1000 1000 1000 1000
* Low hot:  0001 0001 0001 0001 0001 0001 0001 0001 0001 0001 0001 0001 0001 0001 0001 0001

It's possible to derive either of these from the other by shifting. It's
probably worth using a lookup table for one or both of them, especially because
I don't think that I need more than 16 of these bit vectors. (The largest
fixed-size byte arrays that I'm interested in are 16 bytes.)

Then you do this (I was able to coax Claude into using this strategy):

    uint64_t low       = arg & ~high_hot;             // clear top bit
    uint64_t carried   = (low + low_hot) & high_hot;  // top bit set iff low g-1 bits were all 1
    uint64_t high_conj = carried & arg;               // AND back in the real top bit
    return _pext(high_conj, hi);

Here's an entire example that Claude has synthesized:

    #include <stdint.h>
    #include <immintrin.h>   // -mbmi2
    
    // index by g; entry 0 unused (precondition: 1 <= g <= 16)
    static const uint64_t LSB_FOR_G[17] = {
        0x0000000000000000ULL,  // g=0  (unused)
        0xFFFFFFFFFFFFFFFFULL,  // g=1
        0x5555555555555555ULL,  // g=2
        0x1249249249249249ULL,  // g=3
        0x1111111111111111ULL,  // g=4
        0x0084210842108421ULL,  // g=5
        0x0041041041041041ULL,  // g=6
        0x0102040810204081ULL,  // g=7
        0x0101010101010101ULL,  // g=8
        0x0040201008040201ULL,  // g=9
        0x0004010040100401ULL,  // g=10
        0x0000100200400801ULL,  // g=11
        0x0001001001001001ULL,  // g=12
        0x0000008004002001ULL,  // g=13
        0x0000040010004001ULL,  // g=14
        0x0000200040008001ULL,  // g=15
        0x0001000100010001ULL,  // g=16
    };
    
    uint64_t group_bits(uint64_t m, int g) {   // precondition: 1 <= g <= 16
        uint64_t lsb = LSB_FOR_G[g];
        uint64_t hi  = lsb << (g - 1);
    
        uint64_t low      = m & ~hi;
        uint64_t carried  = (low + lsb) & hi;
        uint64_t all_ones = carried & m;
    
        return _pext_u64(all_ones, hi);
    }

This is pretty good. An easy possible improvement is that this is likely
to be called into of a loop. The read from the lookup table could be hoisted
out of the loop.
